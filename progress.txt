CONTEXTMATCH PROJECT PROGRESS
Last updated: June 22, 2026


1. PROJECT OBJECTIVE

The challenge is to rank 100,000 candidate profiles for the supplied job
description and submit the top candidates with factual explanations.

The official leaderboard is hidden during the competition. We therefore cannot
measure our real score until the final results. Our local evaluation must rely
on reproducible proxy evidence, structural analysis, counterfactual tests, and
agreement between independent methods.

The important known evaluation facts are:

- Relevance is graded rather than purely binary.
- The composite is 0.50*NDCG@10 + 0.30*NDCG@50 + 0.15*MAP +
  0.05*P@10.
- Tier 3 or higher is treated as relevant for P@10.
- Honeypots are forced to tier 0.
- A honeypot rate above 10 percent in the submitted top 100 causes Stage-3
  disqualification.
- The documentation mentions a strong plain-language Tier 5 candidate.
- The complete tier definitions and hidden gain mapping are not provided.
- At most three submissions are allowed, and the last valid submission counts.

This means that any locally assigned relevance tier is a hypothesis. It must
not be described as recovered ground truth.


2. GIT AND DATASET STATUS

The original candidates.jsonl file is approximately 464.69 MB. A normal GitHub
push failed because GitHub rejects individual regular Git objects larger than
100 MB.

The large dataset can only be pushed to GitHub by using Git LFS, splitting or
compressing it into smaller files, publishing it as a release asset, or storing
it outside the normal Git repository.

The repository currently has Git LFS-related changes, but the Linux environment
used for analysis does not have the git-lfs executable installed. The dataset
push problem is not considered finished yet. It is separate from the ranking
research and should be resolved before the repository is finalized.

Important: candidates.jsonl must not be accidentally committed again as a
normal 464 MB Git object.


3. WHY MANUAL LABELING WAS REJECTED AS THE PRIMARY METHOD

We discussed manually labeling candidates to create local evaluation data.
Manual labels would have several weaknesses:

- Different reviewers can interpret the same evidence differently.
- One reviewer can change standards during a long labeling session.
- Candidate order creates relativity and anchoring effects.
- Fatigue reduces consistency.
- The strongest candidates are rare, so random labeling is inefficient.
- Synthetic traps can look persuasive to a human reviewer.
- A large labeling exercise is slow and difficult to reproduce.

Manual review may still be useful for a very small disagreement set, but it
should not be the main source of proxy ground truth.


4. THE TWO INDEPENDENT RESEARCH IDEAS

We deliberately separated the work into Idea 1 and Idea 2.


4.1 IDEA 1: UNCONSTRAINED TEACHER

Idea 1 uses the strongest freely available models and as much offline compute
time as necessary to produce a high-quality teacher ranking.

Proposed teacher pipeline:

1. Run deterministic integrity and honeypot checks.
2. Use a strong embedding model for high-recall retrieval.
3. Use a cross-encoder reranker on the shortlist.
4. Use one or more reasoning models for structured candidate evaluation.
5. Generate graded labels, pairwise preferences, evidence, and confidence.
6. Train a compact student ranker for the official runtime constraints.

The teacher is not official ground truth. It is an independent development
oracle that may still contain model errors.

Current Idea 1 status:

- The design is documented in idea1.md.
- The latest public team repository was cloned, reviewed, and then removed from
  this repository to avoid committing nested git history.
- Its final top-100 CSV is preserved as `idea1_top100.csv`.
- It implements a Qwen3-14B-AWQ semantic reranker over 641 shortlisted
  candidates and produces structured assessments, repeat scoring, comparative
  reranking, and a top-100 CSV.
- It does not implement the full independent Idea 1 design: there is no
  full-dataset retrieval, cross-encoder, model ensemble, or constrained
  student.
- Idea 2 was frozen before these teacher results were inspected.
- The review conclusions are consolidated in §5.19 below.


4.2 IDEA 2: REVERSE-ENGINEER THE SYNTHETIC GENERATOR

Idea 2 attempts to recover the process that generated the profiles, then use
the documented rules and job description to formulate explicit hypotheses
about the hidden labels.

Idea 2 is intentionally independent from Idea 1. Until Idea 2 is frozen, it
must not use:

- Teacher judgments
- LLM-generated labels
- Semantic embeddings
- Hidden leaderboard feedback

Idea 2 may use:

- Exact template analysis
- Structured candidate fields
- Deterministic statistics
- The supplied job description
- Documented challenge rules
- Reproducible external chronology facts

After Idea 2 produces a frozen ranking, Idea 1 will be built independently.
Only then will the two rankings be compared. Idea 2 must not be retuned after
seeing Idea 1.


4.3 EVIDENCE TAXONOMY

The project must keep five different evidence types separate:

1. Documented constraints
   External facts supplied by the competition. These include the metric
   formula, Tier-3 relevance threshold, official honeypots being Tier 0, the
   Stage-3 honeypot-rate rule, and the JD's stated requirements and
   disqualifiers.

2. Reconstructed structure
   Observations recovered from the visible dataset, such as template counts,
   static families, field dependencies, and contradiction clusters.

3. JD-to-tier mapping
   Hypotheses about how reconstructed profile evidence maps to hidden
   relevance tiers. These are not ground truth.

4. World stability
   A robustness diagnostic showing whether a ranking survives different
   plausible assumptions. Stability does not make those assumptions true.

5. Teacher output
   An independent model opinion from Idea 1. Agreement can increase practical
   confidence but is not external validation and can share blind spots with
   Idea 2.

Only documented constraints are external facts available before final results.
Reconstructed structure is observed evidence; the remaining three categories
are hypotheses, diagnostics, or model opinions.

Documented-anchor policy:

- Official honeypots are Tier 0, but their candidate IDs are hidden. Our
  detected contradiction records are proxies, not verified official labels.
- A career confined to pure research with no production deployment is an
  explicit hard JD disqualifier.
- Recent LangChain/OpenAI-only experience without substantial earlier
  production ML and not writing production code for 18 months are strong
  conditional negatives. The JD says "probably not," so they should not be
  silently converted into universal Tier-0 rules.
- The Tier-5 note establishes that demonstrated recommendation/product work can
  outweigh missing AI keywords. It does not prove that any entire reconstructed
  cohort is Tier 5.

- 11 general-professional candidates
- 8 general-software candidates

This population is disjoint from the company pre-founding group.

Current interpretation:

This appears to be a second injected trap family, probably designed to catch

5.6 OTHER STRONG CONTRADICTION FAMILIES

We also reconstructed:

- 25 candidates with technology chronology contradictions
- 37 candidates with large duration corruption

Technology chronology is concentrated in applied ML and explicit senior AI

Current interpretation:

These are strong anomaly signals, but their relationship to the official

They should remain independently auditable features instead of being merged

5.8 GENERATOR RECONSTRUCTION

We implemented a dedicated reverse-engineering layer in:

- analysis/generator_reconstruction.py
- artifacts/analysis/generator_manifest.json
- artifacts/analysis/generator_dependencies.json
- artifacts/analysis/generator_assignments.csv
- artifacts/analysis/generator_reconstruction_report.md

This stage does not use summary wording as a clustering input. Instead, it
treats summary templates as generator atoms and clusters them according to what
they emit:

- Title-family distributions
- Current career-template distributions
- Historical career-template distributions
- Industry distributions
- Skill distributions
- Role-count distributions

The system evaluates hierarchical clustering solutions from k=4 through k=12.

The strongest structural elbow occurs at k=7:

- k=6 silhouette: 0.5623
- k=7 silhouette: 0.6737
- Gain: 0.1114

The maximum silhouette occurs at k=12:

- k=12 silhouette: 0.8006

We interpret these as two levels:

- 7 coarse generator families
- 12 fine generator atoms

This avoids mistaking rare individual senior templates for twelve separate
relevance classes.

5.10 GENERATOR DEPENDENCIES

The strongest measured categorical dependencies are:

- Static class -> title family: NMI 0.933
- Static class -> summary archetype: NMI 0.894
- Summary template -> summary archetype: NMI 0.611
- Summary template -> title family: NMI 0.485
- Static class -> current career template: NMI 0.440
- Current career template -> summary archetype: NMI 0.418
- Headline template -> summary archetype: NMI 0.366
- Static class -> industry: NMI 0.159

Important interpretation:

Titles and static family are strongly coupled. Career templates are associated
with family but vary inside each family. Headlines are archetype-pure but have
many variants. Industry is a much weaker class indicator.

The broad general-professional summaries are especially noisy. Some summaries
describe a marketing manager even when the current occupation differs. This
shows that individual text fields can be decorative or generated independently
and should not be trusted without cross-field career evidence.


5.11 BEHAVIORAL STRUCTURE

The mathematical optimum among two through six behavioral clusters is k=2, but
the silhouette is only 0.113.

The strongest separation between these two groups is:

- Missing offer history
- Offer-acceptance value

Other behavioral differences between the two groups are small.

Current interpretation:

There are not two strong natural behavior classes. Behavior should be modeled
as mostly continuous latent factors plus explicit missing-data indicators.
Calling these two groups real candidate types would be misleading.

Static class explains some behavioral variance:

- Search appearances: eta-squared 0.111
- Recruiter saves: eta-squared 0.109
- GitHub score: eta-squared 0.078
- Inactivity: eta-squared 0.070
- Interview completion: eta-squared 0.068
- Profile views: eta-squared 0.060
- Response time: eta-squared 0.058
- Offer acceptance: eta-squared 0.046

This confirms behavioral confounding: better-looking static classes also tend
to receive better generated behavior.

Therefore, raw behavior must not be added to relevance without adjustment. It
would reward a candidate once through static class and again through behavior
that partly encodes the same class.

Behavior policy should depend on cohort size:

- For large families, normalize continuous behavior within family before
  applying a bounded modifier.
- For smaller families, use hierarchical shrinkage toward a pooled baseline.
- For the 8- and 21-candidate senior families, use a near-zero cap because
  within-family estimates are unstable and no behavioral twins exist.
- Behavior may reorder candidates with comparable static relevance, but it
  must not determine the base relevance family.


5.12 DETERMINISTIC JD EVIDENCE CATALOG

We implemented the first bridge from reconstructed generator structure to the
actual job description:

- analysis/jd_evidence_rubric.json
- analysis/jd_evidence_catalog.py
- artifacts/analysis/jd_evidence_catalog.json
- artifacts/analysis/jd_evidence_report.md

The rubric is versioned and separates:

- Direct requirements
- Core role evidence
- Preferred evidence
- Risk contexts
- Compound evidence that must occur within the same career template
- Structured candidate-level overlay rules, introduced in rubric version 1.0.2
- Versioned skill-signal patterns, introduced in rubric version 1.0.4, used
  only by the candidate-level skill overlay

The catalog annotates all 44 career templates with exact matched patterns and
snippets. It then streams all 100,000 candidates and calculates
candidate-weighted evidence prevalence for all 12 fine atoms.

The catalog does not assign relevance tiers or a combined score. A missing
career-text match means "not observed," not "candidate lacks this skill."

Main findings:

- All 8 plain-language senior candidates have evaluated-ranking evidence in at
  least one career template, even though only 12.5 percent have explicit
  production embedding/vector evidence. The rubric was derived from the JD, so
  this is not independent label validation. It demonstrates that the
  deterministic catalog can operationalize the documented Tier-5 anchor
  without requiring fashionable AI keywords.
- The explicit senior AI atoms generally show very high production retrieval,
  vector/hybrid, ranking-evaluation, and end-to-end ownership evidence.
- The 150 applied-ML candidates are heterogeneous:
  - 82.7 percent have evaluated-ranking evidence.
  - 76.7 percent have production delivery plus evaluation depth.
  - 35.3 percent have direct production embedding-retrieval evidence.
  - 35.3 percent have direct production vector/hybrid evidence.
- The 1,000 generic-ML candidates have no direct evaluated-ranking,
  production-embedding-retrieval, or production-vector/hybrid compound in the
  career template library. About 75.1 percent have explicit Python evidence,
  and 32.0 percent have a computer-vision-primary context.
- Data/backend and general-software atoms contain substantial explicit Python
  evidence but no direct ranking/retrieval compounds in their career
  templates.
- Python is not explicitly mentioned in the senior and applied-ML career
  templates. Candidate skills and assessments are therefore essential before
  interpreting career-text absence as a weakness.
- Career-text Python prevalence is not an ordinal quality signal. It is higher
  in generic ML, data/backend, and software cohorts than in the senior cohorts.
  Direct-requirement compounds and demonstrated system ownership must not be
  replaced by a naive count of atomic keyword matches.
- LLM/RAG context appears in several strong applied and senior atoms. It is a
  review context, not an automatic negative; the JD's conditional negative
  depends on recency, shallowness, and absence of earlier production ML.
- Fine-atom differences provide ordering hypotheses, not a finished ordering.
  For example, `fine_atom_01` has broader compound coverage than
  `fine_atom_03`, but candidate recency, skills, integrity, and behavior have
  not yet been overlaid.
- `fine_atom_04` has 100 percent LLM application context and 100 percent on the
  main retrieval/evaluation compounds. This is deep LLM integration in a
  production ranking system, not evidence of the shallow wrapper pattern
  rejected by the JD.
- No career template contains direct open-source, paper, talk, or public
  technical-validation evidence. That requirement must be checked through
  candidate-level signals or other profile fields.

The first audit caught and removed two false positives:

- A customer-support "feedback loop" is not online experimentation.
- The "production side" of brand/packaging work is not technical production
  delivery.

Compound requirements are intentionally not assembled across unrelated jobs.
For example, embeddings in one role and deployment in a different role do not
automatically become "production embeddings-based retrieval."

Rubric version 1.0.2 added a structured `candidate_overlay_rules` block. These
rules are not prose-only reminders anymore; they are schema-checked inputs for
the candidate-level overlay stage:

- Services-only disqualifier: evaluate the entire career; one consulting role
  is insufficient when product-company experience exists.
- Research-only disqualifier: evaluate the entire career and require absence
  of production deployment across all roles.
- Recent shallow-LLM conditional negative: require under-12-month shallow LLM
  work and no substantial earlier production ML/retrieval/ranking evidence.
- Senior-not-coding conditional negative: evaluate hands-on work in the most
  recent 18 months rather than penalizing a senior title by itself.

The contract is enforced in code: `load_rubric()` now rejects a malformed
overlay block, including missing `evaluation_scope`, missing
`automatic_tier_zero`, wrong field types, or an empty `required_inputs` list.
The unit tests include both a positive shape check and a negative malformed
rubric fixture.

Rubric version 1.0.3 records the Tier-A audit fixes. The changes are narrow
phrase-level additions tied to the JD's explicit evaluation language:

- `ranking_evaluation` now catches recruiter engagement metrics, A/B testing
  alongside a ranker, A/B testing infrastructure, live A/B tests, A/B test
  outcomes, and online A/B engagement metrics.
- `retrieval_search` now catches `query understanding`, which appears in a
  plain-language senior search/retrieval template.
- `production_delivery` now uses a narrower serving pattern, e.g. `serving
  10M+ users`, instead of matching only the first digit after `serving`.

These changes added evaluated-ranking evidence to `career_25f609f7c6ec`,
`career_773c6b592e7d`, and `career_f526a9742420`, and retrieval/search
evidence to `career_e8f80c98b493`. No non-Tier-A template gained
`ranking_evaluation` or `retrieval_search` from these additions.

Rubric version 1.0.4 moved the skill overlay regexes out of
`candidate_overlay.py` and into the versioned rubric. This closes the
auditability gap where career evidence was frozen but skill evidence was a
second, unversioned keyword set. `load_rubric()` now validates the
`skill_signal_patterns` block, including regex compilation. Skill evidence
remains firewalled from career-template compounds: it can fill candidate-level
gaps or expose keyword-stuffing, but it does not create production/retrieval
career evidence by itself.

The skill patterns were also tightened. In particular, `retrieval_search` no
longer matches bare `search`; it requires explicit retrieval/vector/semantic
search wording, BM25, or exact `RAG`. This prevents ordinary skills such as
`Job Search` from leaking into retrieval evidence.


5.13 RUBRIC AUDIT WORKSHEET

Because every one of the 300,171 career entries is a copy of one of only 44
career templates, the rubric's regex behavior is fully determined by a finite,
enumerable input space. A complete human read of all 44 templates therefore
makes extraction errors enumerable and correctable across the visible
career-template library, which is normally impossible for text matching at this
scale. It does not make the extractor infallible; the result is still subject
to human and rubric judgment.

We added a reproducible layout tool:

- analysis/build_audit_worksheet.py
- artifacts/analysis/jd_evidence_audit_worksheet.md

The tool changes no annotations. It lays out all 44 templates with full text,
assigned signals by category, exact matched terms, a snippet, occurrence counts,
and verdict lines, ordered by ranking decisiveness:

- Tier A: senior and applied-ML templates. Verify every signal and every
  absence; one error can materially change a top-cohort candidate, and these
  templates can decide NDCG@10/@50.
- Tier C: generic-ML, the borderline Tier-3 region.
- Tier B: high-population non-ML templates. Confirm only that no false
  direct-requirement match leaks in; false negatives are lower priority but
  should still be noted for tail-exception audits.

It is intentionally not wired into run_all; it is a one-time audit aid, not part
of the analysis pipeline.

Reassuring finding from the first scan: no non-ML template fires any
direct-requirement compound. The keyword-trap risk of a stuffer template
manufacturing top-cohort evidence is therefore not observed in the current
catalog. The only non-ML hits are isolated core-role signals (product context,
production delivery, operational ownership) on legitimately product/engineering
roles, and they do not change the current top-cohort evidence interpretation.
Two judgment calls were noted for the manual pass: whether a bare
"Shipped"/"deployment" should count as production delivery in a data/backend
template, and whether one data/backend template that lights up product,
ownership, and delivery together is fully supported by its text.

Audit status: the full 44-template review is complete. The Tier-A pass produced
rubric version 1.0.3 and the four objective fixes listed in §5.12. Rubric
version 1.0.4 moved candidate skill patterns into the audited schema.

The final Tier-B/C freeze review produced rubric version 1.0.5 with two narrow
false-positive removals:

- `career_9ab513003702` no longer receives `production_delivery` merely from
  `model-serving`; another team built the service and this candidate owned
  integration and observability.
- `career_03ab1210df1d` no longer receives `operational_ownership` from
  `productionization` inside a sentence saying the role was modeling rather
  than productionization.

The three previously open production-delivery judgments are resolved:

- `career_2e516f229493`: accepted because predictive features were explicitly
  shipped.
- `career_8f5eedb01688`: accepted because the SaaS application has explicit
  Docker/Kubernetes deployment.
- `career_9ab513003702`: rejected as delivery ownership.

Tier C was reviewed in both directions across all six generic-ML templates.
Tier B was reviewed for direct-requirement leakage and broad false positives.
No non-ML template creates a direct retrieval/ranking compound. The Tier-B/C
audit is signed off for the pre-teacher Idea 2 freeze at rubric version 1.0.5.

The overlay still does not assign relevance tiers or a score. It produces the

Current overlay results:

- Candidates processed: 100,000
- Rubric version: 1.0.5
- services_only_entire_career fired: 5,345 candidates
- recent_shallow_llm_only fired: 1,868 candidates
- research_only_without_production fired: 2 candidates
- senior_not_coding_recently fired: 428 candidates
- Integrity join recovered the same 162 high-risk candidates, including
  83 company_pre_founding and 21 expert_zero_duration_3plus proxy honeypot
  records.
- Career-level evaluated-ranking compounds are now inherited by 153 candidates,
  up from 129 before the Tier-A audit fixes. `shipper_with_evaluation_depth`
  is inherited by 141 candidates, up from 113.

Important correction from the first overlay run: a draft version treated any
`Manager` title as senior-coded and fired `senior_not_coding_recently` on
22,363 candidates. That was too broad and was fixed. The rule now requires
senior/leadership wording or a technical-manager title, plus absence of recent
hands-on engineering evidence.

Follow-up correction from the senior-rule audit: the hands-on guard no longer
uses title text alone as hands-on evidence. It requires explicit career signals
or production-delivery evidence. After this change,
`senior_not_coding_recently` fires on 428 data/backend-adjacent candidates and
zero candidates in the 8-person plain-language senior cohort or the 21-person
explicit senior-AI cohort. The 29 senior candidates still deserve manual
Tier-A review, but the rule is no longer structurally exempting every
`Senior AI Engineer` by title alone.

Residual senior-rule risk: the 428 firings are concentrated entirely in the
5,000-candidate data/backend-adjacent cohort, mostly `Senior Data Engineer` and
`Senior Software Engineer` records whose templates lack the narrow hands-on
signals. This is probably mild over-firing and low stakes for the main cutoff,
but any data/backend tail exception should be manually checked so the bounded
penalty does not wrongly suppress a genuinely hands-on candidate.

The top-29 senior candidates are not title-auto-cleared in the current code.
Their `senior_not_coding_recently` verdicts are guarded by recent hands-on
career evidence such as production delivery, retrieval/search, ranking, or
embedding signals. The Tier-A manual audit still needs to confirm that these
templates really describe hands-on building, because this rule should not be
the only protection against a non-coding senior architect entering the top
ranked set.

Follow-up correction from the services/research spot-check: the
`research_only_without_production` implementation no longer matches the bare
word `research`. A content/SEO template containing "using LLM tools for
research, drafting, and editing" was a false positive. The rule now requires
explicit `research_only_context`, research-scientist style titles, academic or
research-lab context, or direct "research-only"/"pure research" wording. This
reduced firings from 1,418 to 2. Services-only has zero product-context or
product-industry guard leaks in the current overlay; 97 services-only records
also show production-delivery evidence, so this rule should remain a bounded
services-career penalty rather than an automatic tier-zero kill.


5.15 FIRST IDEA 2 MULTI-WORLD SCORER

We implemented the first deterministic Idea 2 proxy scorer in:

6. CURRENT RECONSTRUCTED GENERATOR GRAPH

The best current model of the data-generation process is:

Latent static profile family
    |
    +-- summary template
    +-- headline template
    +-- title family and current title
    +-- current and historical career templates
    +-- industry and company distributions
    +-- skill distribution
    +-- experience and role-count distributions
    |
    +-- behavior distribution conditioned partly on profile family
    |
    +-- optional rare post-generation contradiction mutation
            |
            +-- company chronology mutation
            +-- technology chronology mutation
            +-- duration corruption
            +-- zero-duration expert mutation

This dependency graph is the actual reverse engineering in Idea 2. It goes
beyond manually writing a relevance rubric: it attempts to identify the latent
generator families, their emitted fields, and the mechanisms that alter a small
number of records after normal profile generation.

8. MAIN RISKS


8.1 ARCHETYPE IS NOT AUTOMATICALLY RELEVANCE

Recovering generator classes does not prove their official relevance grades.
The generator may create profile classes first and apply additional label rules
later.


8.2 BEHAVIORAL DOUBLE COUNTING

Behavior improves with static class. A ranker that uses both without
deconfounding can count the same latent quality twice.


8.3 TOP-TIER BEHAVIOR HAS NO TWIN COVERAGE

There are no behavioral twins for the senior cohorts. Their behavioral
adjustment must remain conservative and uncertain.


8.4 KNOWLEDGE-BASE COVERAGE

Chronology detection currently depends on a small, versioned list of companies
and technologies. Unknown young companies can create false negatives.


8.5 GENERATOR OVERFITTING

Exact template reconstruction may score this synthetic dataset well while
failing on real or newly worded profiles. The final constrained student should
learn general career evidence rather than candidate IDs or exact template IDs.


8.6 HIDDEN LABEL RULES

The organizers may have applied rules, randomness, or manual adjustments that
are not recoverable from visible data alone.

- What IS strongly supported: the generator has discrete, reproducible profile
  families, recoverable from emitted fields alone, at stable population sizes
  (8 / 21 / 150 / 1,000 / 5,000 / 25,000 / 68,821).
- What is NOT validated: that these families map to relevance tiers, or in what
  order. Every emitted field co-varies with the latent class by construction, so
  high cross-field agreement is partly guaranteed and says nothing about the
  hidden label ordering.

Conclusion: the seven-family recovery is well supported and reproducible,
while the 12-atom hierarchy preserves granularity uncertainty. The
class-to-tier ordinal mapping remains 100 percent hypothesis. This is the
single most important thing not to lose sight of. Idea 1 can independently
stress-test this ordering, but neither teacher agreement nor world stability
can validate it as ground truth. Only the documented anchors are external
constraints.


A.2 THE SCORING MATH JUSTIFIES A HIGH-PRIORITY SHORTLIST, NOT A KNOWN RELEVANT POOL

Composite = 0.50*NDCG@10 + 0.30*NDCG@50 + 0.15*MAP + 0.05*P@10. Eighty percent
of the score lives in the top 50, and half the score lives in the top 10.

The strongest reconstructed cohorts total 179 candidates (8 + 21 + 150), with
generic ML adding 1,000 candidates. Together these 1,179 records are a sensible
high-priority shortlist designed for high recall. Its recall has not yet been
measured. It is not a known relevant pool, and the remaining candidates cannot
be assumed irrelevant before checking for career-supported exceptions in
software, data/backend, and direct occupations.

Strategic consequences:

- Expensive reasoning should concentrate on this shortlist after a high-recall
  retrieval pass over all 100,000 candidates.
- Idea 1 should retrieve broadly over the full dataset, then apply expensive
  teacher judgments to approximately 1,200 candidates plus tail-audit samples.
- The 29 senior candidates deserve disproportionate attention, but they are not
  guaranteed to determine NDCG@10. Applied-ML candidates or exceptions from
  broader cohorts may outrank some seniors under the hidden labels.
- The internal ordering of the low-priority tail is lower value, but recall
  audits are still required before excluding it from expensive analysis.


A.3 URGENT: A NAIVE STRUCTURE-PRIOR RANKER COULD BE DISQUALIFIED

Honeypot rate above 10 percent in the top 100 is an automatic Stage-3
disqualification. I verified that 108 high-confidence-contradiction candidates
sit inside the three high-priority ML-looking cohorts:

- senior_explicit_ai: 8
- applied_ml: 29
- generic_ml: 71

These are precisely the records a static-family prior would push toward the top.
A ranker that orders by reconstructed class and ignores the integrity engine
could pull more than 10 contradiction-flagged records into its top 100 and risk
disqualification before scoring even matters. The exact count depends on the
within-family ordering, so disqualification is a serious risk rather than a
mathematical certainty.

Therefore the integrity engine is not polish. It is survival. Integrity
auditing must run before final selection, and the top 100 must be clean under
the chosen versioned integrity policy. A conservative submission policy is
zero company-pre-founding or three-plus zero-duration-expert records in the
final top 100, explicit review of technology and duration flags, and a logged
proxy-honeypot check as the last gate. This does not prove the official hidden
honeypot rate is zero because the official IDs remain hidden.


A.4 BEHAVIOR NORMALIZATION BREAKS EXACTLY WHERE IT MATTERS MOST

Correction to an earlier overstatement: the static_class -> behavior_cluster
NMI of 0.020 only says the WEAK two-cluster discretization of behavior is nearly
independent of class. It does NOT make raw behavior independent of class. The
operative measures are the median gradient (response rate rises 0.42 -> 0.79 up
the family ladder) and the eta-squared values (up to ~0.11 for search
appearances and recruiter saves). Roughly a tenth of the variance in some
behavior features is explained by static class, so raw behavior leaks class
moderately, and within-class normalization plus a bounded modifier IS justified.

The catch is granularity-dependent. Within-class normalization is sound for
large families such as generic ML and the larger adjacent classes. Applied ML
may support shrinkage-assisted estimates. The 8- and 21-candidate senior
families are too small to estimate stable within-family behavior distributions,
and they have no twins.

Recommendation: normalize behavior within family only where N is large enough.
For smaller cohorts, use hierarchical shrinkage toward an appropriate pooled
baseline. Apply a near-zero, explicit cap to the senior families. Behavior
should break ties among statically comparable candidates, not decide base
relevance or guarantee a senior a top-10 position.


A.5 THE EVIDENCE IS A PROPERTY OF TEMPLATES, NOT CANDIDATES

The compactness of the generator makes a fully independent per-candidate
semantic pass unnecessary. There are only 44 career templates and 12 fine
atoms. Evidence should be characterized once per template/atom, and candidates
inherit it through the templates they carry, with candidate-specific overlays
only for dates, companies, behavior, and mutations. The main plan above has
been reordered accordingly: characterize the library first, then build the
candidate overlay.


A.6 INDEPENDENCE LEAK TO GUARD IN THE EVIDENCE CATALOG

Writing the human-readable semantic description of each career template/atom is
allowed within Idea 2 only if it is done by deterministic rules or by a human
reading the text. An LLM-generated candidate judgment, semantic score, or
template-to-tier decision would import teacher-like evidence into Idea 2 and
break the intended comparison. AI coding assistance is not itself leakage if
the resulting rules are explicit, deterministic, reviewed, and reproducible.
Keyword/regex rules over the 44 templates are acceptable.


A.7 THE UNCERTAINTY-WORLDS EVALUATOR IS A ROBUSTNESS HARNESS, NOT A LABEL ORACLE

Because there is no leaderboard and only three submissions, multi-world
evaluation should be a central development harness. It is not the only
pre-submission signal: documented anchors, integrity checks, counterfactual
tests, twin analysis, retrieval audits, and the independent teacher also
provide evidence or diagnostics.

Judge each ranker by mean, worst-case, and variance across worlds, plus
top-10/top-50 stability. But remember that all worlds are constructed from our
own assumptions. A ranker can be stable across a poorly chosen world set and
still be wrong.

A built-in regression test for the eventual scorer: the documented honeypot
example ("8 years at a company founded 3 years ago") means the integrity gate
must reject a synthetic fixture with that contradiction. A conservative Idea 2
policy may hard-exclude the 83 detected company-pre-founding records, but tests
should validate the rule behavior rather than claim that all 83 IDs are known
official Tier-0 labels.

A.9 MY BOTTOM LINE

The Idea 2 reconstruction is better than I expected when we started, and the
discipline (no teacher leakage, hedged language, reproducible artifacts, tests)
is exactly what survives the Stage 3-5 human review. The structure half is
essentially done.

The remaining risk is no longer technical recovery. It is two things: (1) the
unvalidated class-to-tier ordering, which documented anchors constrain and the
independent teacher plus multi-world evaluation can only stress-test, and (2)
forgetting that integrity filtering is a disqualification gate, not merely a
feature. If the next stage produces a frozen, integrity-gated,
multi-world-evaluated probabilistic ranking over a shortlist designed for high
recall, with audits of the excluded tail, Idea 2 will have done its job well.


A.10 CURRENT SUBMISSION-PATH SOLUTION STATE

The submission-facing `solution/` path is now a general JD-to-candidate ranker,
not an Idea 1 or Idea 2 artifact. It parses the JD, builds a structured
requirement spec, scores precomputed candidate features, hard-gates integrity
and disqualifier failures, enforces a career-evidence floor, then ranks by a
career-evidence-dominant score with bounded semantic, skill, seniority,
behavior, logistics, and soft-negative terms.

The latest scoring change adopted the senior-ordering experiment into
`solution/ranker.py`. This is not a hardcoded senior boost. It reuses the
existing JD-derived `seniority_alignment` signal, normalizes positive fit by the
JD's seniority strength, and applies a capped ordering term of 0.4. It only
affects candidates who already meet the career-evidence floor; it cannot admit
skills-only, semantic-only, non-technical, or integrity-blocked profiles.

Why it was adopted:

- Top-100 membership stayed unchanged relative to the previous solution.
- Top-10 composition moved from 5 applied / 3 plain senior / 2 explicit senior
  to 4 applied / 4 plain senior / 2 explicit senior.
- The main output now has all 8 plain-language seniors in the top 100 and keeps
  13 of 21 explicit-senior-AI candidates.
- Dataset audit is clean: 0 unsafe top-100 candidates and 0 direct-occupation
  leakage.
- It improved the local proxy composite under all tested truth/gain assumptions:
  Idea1-top100 proxy, Idea2-full-tier proxy, and Idea2-top100 proxy, each under
  exponential and linear gains.

Current validated output:

- `artifacts/analysis/solution_ranker_submission.csv`
- `solution.validate_output`: passed for 100 rows against 100,000 candidates.
- Organizer validator: "Submission is valid."
- Runtime using precomputed features: about 18 seconds, about 18 MB max RSS.
- Full solution test suite: 41 passed.

The remaining caveat is unchanged: proxy scores are diagnostics, not official
score estimates. The local proxy composite now ranges roughly 0.642 to 0.877
depending on truth assumption and gain mapping, but the real hidden labels are
unknown. The useful interpretation is relative: the adopted senior-ordering
change improved every local proxy without weakening safety gates.


A.11 FINAL ROLE-TITLE ORDERING STEP

After the seniority-ordering adoption, the weak remaining metric was NDCG@10.
The top-100 set was already clean and stable, but the first two ranks were still
applied-ML profiles under the local proxy tiers. A final bounded experiment
tested an additional JD-derived role-title ordering term:

- only for senior/staff/principal IC-style JDs;
- only for candidates already passing the career-evidence floor;
- requires demonstrated ownership/scope in career evidence;
- requires explicit senior/staff/lead/principal technical title alignment;
- scores title-family/function fit from the parsed JD role title;
- cap = 0.4, so it cannot overpower the evidence score or admit weak profiles.

This is still a general JD feature, not an Idea 1/Idea 2 cohort boost. It uses
the JD title and candidate title/evidence only.

The experiment was adopted because it improved every local proxy cell while
preserving all safety gates:

- Top-100 membership unchanged: 100/100 common with the previous solution.
- Top-100 audit unchanged: 0 unsafe, 0 direct-occupation leakage.
- Top-10 mix became 4 plain senior, 3 explicit senior, 3 applied ML.
- Proxy composite range improved to roughly 0.748 to 0.925.
- NDCG@10 improved to 0.780 under exponential gain and 0.909 under linear gain.
- Main output and experiment output are identical after adoption.
